## 概要

タスク追加・管理操作のUXを改善する5つの変更をまとめて実施する。
キーボード操作の効率化・メモ記入の場の追加・誤操作防止・時間/期限の入力・表示フォーマット改善が目的。

**スコープ**
- やること: Ctrl+Enter送信 / 詳細フィールド追加 / 削除確認ダイアログ / 見積もり時間を分単位化 / 期限をMMDD形式で表示・入力
- やらないこと: 既存データの自動移行処理（リリース前のため移行性は不要）

## 背景・課題

- **Ctrl+Enter**: タスク追加モーダルでマウスに持ち替えて「保存」を押す手間がある。キーボードで完結させたい
- **詳細フィールド**: タイトルだけではメモしきれない内容（背景・手順・リンクなど）をタスクに紐づけたい。現状タスク外で管理するしかない
- **削除確認**: 誤削除を防ぐ仕組みがなく、一度削除すると取り消せないためリスクがある
- **見積もり時間**: 現在は時間単位・0.5刻みの数値入力だが、数値を直打ちするほうが速い。また15分・30分など1時間未満のタスクも多い
- **期限表示**: `2026-06-14` という表示は冗長。短期タスクが中心のため年は不要で `06/14` のようなMMDD形式が読みやすい

## 要件

### 機能要件

**Ctrl+Enter でタスク保存**
- タスク追加モーダル・編集モーダルの両方で Ctrl+Enter を押すとフォームを送信する
- 動作は保存ボタンのクリックと同等

**タスク詳細フィールド**
- タスクに `description`（詳細）フィールドを追加する
- 入力: テキストエリア、任意入力（空でもOK）
- 表示: タスク一覧カードでは非表示。編集モーダルを開いたときのみ表示・編集可能

**削除確認ダイアログ**
- 削除ボタン押下時にブラウザ標準の `confirm()` ダイアログを表示する
- OKで削除実行、キャンセルで何もしない

**見積もり時間を分単位に変更**
- バックエンドのフィールド名を `estimate_hours`（float）から `estimate_minutes`（int）に変更する
- モーダル入力: 分の整数値を直接入力（例: `90`）
- 一覧表示: `H:MM` 形式で表示（例: `1:30`、`0:05`）
- 最小値: 1分以上

**期限の入力・表示をMMDD形式に**
- 入力: テキスト入力（`0614` と打つと `2026-06-14` として保存）＋カレンダーピッカーアイコンを横に添える
- 表示: `MM/DD` 形式（例: `06/14`）
- 年はUI上で扱わない（保存時は現在年を補完）
- バリデーション: 不正な日付（`1399` など存在しない日付）は入力エラーとして弾く

### 非機能要件

- 特になし（個人利用のため）

## 検討した選択肢

### 期限の入力方式

#### 選択肢A: テキスト入力 + カレンダーピッカーを横並び（別フィールド）
- メリット: 実装がシンプル
- デメリット: UIが重複した見た目になる

#### 選択肢B: type="text" にして MMDD 入力をパース、ピッカーはアイコンとして添える（採用）
- メリット: `0614` と打つだけで完結し、キーボード操作が速い
- デメリット: バリデーション（不正な日付）を自前で実装する必要がある

### 見積もり時間のフィールド変更方法

#### 選択肢A: バックエンドのフィールド名を `estimate_minutes`（int）に変更（採用）
- メリット: データの意味と型が一致する
- デメリット: 既存の `tasks.json` の `estimate_hours` フィールドとの互換性が崩れる

#### 選択肢B: バックエンドは `estimate_hours`（float）のまま、フロントで変換
- メリット: バックエンド・既存データへの影響ゼロ
- デメリット: フィールド名と実態（分）がずれ、将来的に混乱を招く

## 採用した方針と理由

- **期限入力**: 選択肢B（テキスト入力 + ピッカーアイコン）を採用。キーボードで完結する操作性を優先
- **見積もり時間**: 選択肢A（フィールド名ごと `estimate_minutes` に変更）を採用。データとフィールド名の整合性を優先。リリース前のため既存データの移行性は考慮しない

## この決定によって生じる影響

**良い影響**
- キーボードだけでタスク追加が完結し、操作が速くなる
- タスクに詳細メモを紐づけられるようにより使いやすいタスク管理が可能になる
- 誤削除リスクが減る
- 時間・期限のフォーマットが実態に即したものになり、読みやすくなる

**悪い影響・トレードオフ**
- `estimate_hours` → `estimate_minutes` の変更により、既存の `tasks.json` データは手動で修正が必要（リリース前のため許容）
- 期限のテキスト入力バリデーションを自前で実装するため、実装コストがやや増す
- Ctrl+Enter のグローバルショートカットは `window` に keydown リスナーを登録するため、モーダル外でも Ctrl+Enter が発火する点に注意（モーダルが開いている間のみ登録・解除されるため実害なし）
- `useCallback` + `useEffect` でのリスナー管理において Rules of Hooks 違反（アーリーリターン後のフック呼び出し）が発生していたため、`useRef` を使った設計に変更した。`form` / `dueDateInput` / `submitting` を ref 経由で参照することでリスナーの再登録を最小化している

## 仕様変更の経緯

- 期限入力の年補完ロジック: 仕様では「現在年を補完」としていたが、現在月より前の月を入力した場合に過去日付が保存される問題があるため、現在月より前の月入力は翌年として補完する実装を採用した。
- Ctrl+Enter の実装: 当初 `useCallback` の依存配列に `form` 等を含めていたが、フォーム入力のたびにグローバルリスナーが再登録される問題があった。最終的に `useRef` パターンで最新値を参照する設計に変更した。
