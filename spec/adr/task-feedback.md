# タスクへのフィードバック機能

## 概要

タスクに対してフリーテキストのフィードバックコメントを残せる機能。
蓄積されたFBは、ユーザーが明示的に指示したタイミングでClaudeが読み込み、`score-prompt.md` 自体を改善するための材料として活用する。
スコアリング実行時には自動でFBを参照しない。

## 背景・課題

- スコアリングの精度はプロンプトの質に依存するが、現時点でユーザーが結果に違和感を覚えても改善を記録する手段がない
- 「このタスクのスコアが高すぎる」「気分が悪いときはこういう軽いタスクを優先してほしい」など、使いながら気づいた知見をその場で残せる場所がない
- FBをプロンプトに都度差し込む方式だと件数が増えるにつれプロンプトが肥大化するため、採用しない

## 要件

### 機能要件

- タスクカードの `task-actions` 列に吹き出しアイコンボタン（💬）を追加する
  - **タスクカードの高さは変えない**（既存の編集・完了ボタンと同じ行に収める）
- ボタンを押すと `FeedbackModal` が開き、テキストエリアでコメントを入力できる
- 送信すると `POST /tasks/{task_id}/feedback` を呼び出し、`backend/feedback.json` に追記保存される
- FBの表示・編集・削除は今回のスコープ外

### データ形式（`backend/feedback.json`）

```json
[
  { "task_id": "xxx", "title": "タスク名", "comment": "スコアが高すぎる", "created_at": "2026-06-20" }
]
```

### APIエンドポイント

- `POST /tasks/{task_id}/feedback` — `{ comment: str }` を受け取り `feedback.json` に追記
- `GET /tasks/{task_id}/feedback` はバックエンドに実装するが、フロントからは今回使用しない

### FBのプロンプト差し込みは行わない

- スコアリング実行時（`/tasks/score/run`）に `feedback.json` を読み込む処理は追加しない
- FBはユーザーが明示的に「FBを見てプロンプトを改善して」と指示したときにClaudeが手動で読み込み、`score-prompt.md` を書き換える材料として使う

### エッジケース

- `feedback.json` が存在しない場合は初回書き込み時に新規作成する
- `comment` が空の場合はバリデーションエラーを返す

## 検討した選択肢

### 選択肢A: FBをスコアリング実行時にプロンプトへ自動差し込み（issue初期案）

- メリット: FBが即座にスコアに反映される
- デメリット: FBが蓄積するほどプロンプトが肥大化し、コストと精度に悪影響が出る可能性がある

### 選択肢B: FBは保存のみ、プロンプト改善はユーザー指示時にClaudeが手動実行（採用）

- メリット: プロンプト肥大化を防げる。ユーザーが意図的にタイミングを選んでプロンプトを育てられる
- デメリット: FBを書いてもすぐにスコアへ反映されるわけではない（ユーザーが改善指示を出す必要がある）

## 採用した方針と理由

**選択肢Bを採用。**

FBをスコアリングプロンプトに都度差し込む方式は件数増加とともにプロンプトが肥大化するため採用しない。
FBはユーザーが「FBを見てプロンプトを改善して」と明示的に指示したときにのみClaudeが読み込み、`score-prompt.md` 自体を書き換える材料として使う運用とする。

## 状態

完了（2026-06-20）

## この決定によって生じる影響

### 良い影響

- プロンプトが肥大化しないためスコアリングのコスト・精度が安定する
- ユーザーがFBを見直して「本当に反映すべき知見」だけをプロンプトに取り込めるため、プロンプトの質が上がりやすい

### 悪い影響・トレードオフ

- FBを書いても自動でスコアに反映されないため、ユーザーが意識的に「プロンプト改善」の指示を出す必要がある
- プロンプト改善フローが明文化されていないため、FBが溜まりっぱなしになるリスクがある（将来的に改善フロー整備が必要）

## 仕様変更の経緯

### 2026-06-20: FBのスコアリングプロンプトへの自動差し込みを撤廃

- 変更前: `/tasks/score/run` 実行時に `feedback.json` を読み込み、プロンプト末尾に差し込む（issue初期案）
- 変更理由: FBが蓄積するとプロンプトが肥大化するため。FBはプロンプト自体を改善する材料として使う設計に変更
- 変更後: FBは保存のみ。ユーザーが明示的に指示したときにClaudeが読み込んで `score-prompt.md` を改善する

### 2026-06-20: 💬ボタンの配置をタスクカードから編集モーダルへ変更

- 変更前: タスクカードの `task-actions` 列に💬ボタンを配置（issue初期案）
- 変更理由: タスクカードに追加するとカードの高さが増えてしまい、UIの設計ルール（カードの高さを変えない）に違反した
- 変更後: 編集モーダルのヘッダー（`modal-header-actions`）に💬ボタンを配置

### 2026-06-20: FBのデータ形式をタスク属性の全スナップショットに拡張

- 変更前: `{ task_id, title, comment, created_at }` のみ保存
- 変更理由: タスクを後から編集すると属性が変わり、FB投稿時点の状態が失われる。FBはプロンプト改善の材料として使うため、投稿時点のコンテキストを完全に保持すべきと判断
- 変更後: `estimate_size`, `bother_level`, `due_date`, `importance`, `description` を追加したスナップショット形式で保存
