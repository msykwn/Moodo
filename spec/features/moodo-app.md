## 概要

仕事のタスクに「作業見積もり・めんどくさレベル・期限・重要度」を設定し、「今の気分・空き時間」をもとにAIがおすすめスコアを算出して表示するWebアプリ。ただのTODOリストではなく、そのときの気分や状況を踏まえてタスクを選べるようにすることで、消化作業にならず自分の気分を優先した人間らしい働き方を支援する。

**スコープ**
- やること: タスク管理(CRUD) / 気分・空き時間の入力 / AIによるおすすめスコアの算出と表示
- やらないこと: カレンダー連携 / チーム共有 / 繰り返しタスク / データエクスポート/インポート機能 / Claude APIのコード組み込み

## 背景・課題

- 既存のタスク管理ツール(Todoistなど)はタスクを羅列するだけで、使いこなしている実感がなかった。タイトルだけ並んでいてもメモ帳と変わらない
- 期限順にただこなすだけでは消化作業になり、仕事が楽しくなくなる
- タスクの所要時間・難易度・期限などの属性を見える化し、そのときの気分や空き時間を踏まえて「自分で選べる」状態にしたい
- 根本の動機: 仕事を楽しくしたい。これがコンセプト「人間らしく働く」に直結している

## 要件

### 機能要件

**タスク管理**
- タスクのCRUD（追加・編集・削除）。追加/編集はモーダルで行う
- タスク属性:
  - タイトル（テキスト）
  - 作業見積もり（時間）
  - めんどくさレベル（楽勝 / 普通 / めんどう / やりたくない）
  - 期限（日付）
  - 重要度（低 / 中 / 高）

**気分・空き時間の入力**
- 画面左上に常時表示される入力欄
- 前回値を引き継ぎ、いつでも変更可能
- 変更時はAI評価を再実行する

**AI評価・おすすめスコア表示**
- アプリ起動時にAI評価を実行してスコアを算出する
- 評価フロー:
  1. アプリがタスクデータ＋気分・空き時間をJSONとして書き出す（エクスポート）
  2. ユーザーがそのJSONをClaude.aiに渡してスコア評価を依頼
  3. ClaudeがスコアをつけたJSONを返す
  4. ユーザーがそのJSONをアプリに読み込ませると、おすすめスコアが反映される
- おすすめスコア(0〜100%)を各タスクに表示
- スコアに応じて色分け:
  - 高（70%以上）: 緑
  - 中（40〜69%）: オレンジ
  - 低（39%以下）: グレー
- 色の閾値は後で調整できる構成にする

**タスク一覧**
- 期限順で表示
- タスク0件時・AI評価中のローディング表示は実装者に委ねる

**データ保存**
- `tasks.json` ファイルをバックエンドが読み書きする
- localStorageは使用しない

### 非機能要件

- 特になし（個人利用のため）

## 検討した選択肢

### AI評価の実装方法

#### 選択肢A: システムプロンプトにルールを書きClaudeに判断させる（採用）
- メリット: ルールを自然言語で柔軟に記述できる。「気分が重いときはめんどくさレベルが低いタスクを優先」のような複合条件も書きやすい
- デメリット: Claudeの判断がブレることがある。プロンプトの品質に依存する

#### 選択肢B: コード側でスコアを計算し、Claudeには最終コメント生成だけ頼む
- メリット: スコア計算が安定・高速。AI呼び出しコストを抑えられる
- デメリット: ルールをコードで書く必要があり、複合条件が増えると複雑になる

#### 選択肢C: コード側でルールスコアを計算し、それをClaudeに渡して最終スコアを出させる
- メリット: ルールの安定性とAIの柔軟な判断を組み合わせられる
- デメリット: 実装がやや複雑

### データ保存・ファイル読み書きの方法

#### 選択肢A: ファイル選択UIで読み込み・ダウンロードで書き出し
- メリット: バックエンド不要
- デメリット: 毎回ダウンロードフォルダに保存されて煩わしい。アプリ上の編集がリアルタイムで反映されない

#### 選択肢B: 薄いバックエンド(FastAPI)を追加してファイルを読み書き（採用）
- メリット: アプリ上の編集が即座に `tasks.json` に反映。AI評価用のJSONエクスポート/インポートもAPIで扱える
- デメリット: 起動が2コマンド必要（`npm run dev` + `uvicorn`）

#### 選択肢C: Electron化
- メリット: デスクトップアプリとして1クリック起動
- デメリット: セットアップが重くなる

### バックエンドのフレームワーク

#### Hono (Node.js)
- メリット: Node.js1本で完結。TypeScriptでフロントと統一できる。軽量
- デメリット: 比較的新しく情報量がExpressより少ない

#### Express (Node.js)
- メリット: 最も事例が多く情報が豊富
- デメリット: Honoより冗長

#### FastAPI (Python)（採用）
- メリット: NBAプロジェクトと同じPython/uvの構成で慣れている。型定義が自動でAPIドキュメントも自動生成される
- デメリット: Node.jsとPythonの2ランタイムが必要。Windowsでの環境構築が少し手間

## 採用した方針と理由

- **フロントエンド**: React (Vite) + TypeScript
- **バックエンド**: FastAPI (Python) + uv
- **データ保存**: `tasks.json` ファイル（FastAPIが読み書き）
- **AI評価**: システムプロンプトにルールを書き、タスクデータをJSON渡しでClaudeにスコアを返させる。Claude APIのコード組み込みはせず、Claude.aiへの手動貼り付けで運用する
- **起動**: `npm run dev`（フロント）+ `uvicorn`（バック）の2コマンド

FastAPIを採用した理由: 既存のNBAプロジェクトと同じPython/uv構成のため環境に慣れており、学習コストが低い。今回AIとのコード連携は不要なため、Node.js統一のメリットより既存環境の活用を優先した。

## この決定によって生じる影響

**良い影響**
- タスクの属性（見積もり・めんどくさレベル・期限・重要度）が見える化されることで、タスク選択の判断材料が増える
- 気分・空き時間を加味したAI推薦により、消化作業ではなく能動的なタスク選択ができる
- 仕事を楽しくする体験を作れる

**悪い影響・トレードオフ**
- `tasks.json` が壊れたときにデータが全消えするリスクがある（バックアップなし）
- 起動が2コマンド必要で少し手間
- AI評価がClaude.aiへの手動貼り付け→JSONインポートという手動フローのため、評価の手軽さはやや低い
